# Auto-Generate EventPayload TypeScript Types — Design Spec

**Date:** 2026-05-04  
**Status:** Approved

## Problem

Rust `EventPayload` enum (24 variants) and related domain types are manually mirrored as TypeScript types in `apps/agent-gui/src/types/index.ts`. Every new event variant requires coordinated edits across Rust and TypeScript, with three consumer files (`session.ts`, `useTauriEvents.ts`, `useTraceStore.ts`) using unsafe `as` type assertions. The current `scripts/check-types.sh` only validates variant name parity via string matching — it cannot detect field name or type mismatches.

## Goal

Auto-generate TypeScript types for `EventPayload`, `DomainEvent`, and related domain enums/structs from Rust via specta, refactor consumers to use TypeScript discriminated union narrowing (removing `as` assertions), and replace the fragile `check-types.sh` with a generation + diff check.

## Scope

### Generated types (from Rust via specta)

| Rust Type               | Crate        | TypeScript Output                   |
| ----------------------- | ------------ | ----------------------------------- |
| `EventPayload`          | agent-core   | discriminated union with `type` tag |
| `DomainEvent`           | agent-core   | interface containing `EventPayload` |
| `PrivacyClassification` | agent-core   | string union                        |
| `AgentRole`             | agent-core   | string union                        |
| `TaskState`             | agent-core   | string union                        |
| `TaskSnapshot`          | agent-core   | interface                           |
| `TaskGraphSnapshot`     | agent-core   | interface                           |
| `MemoryScope`           | agent-memory | string union                        |

### Not generated (remain hand-written)

- `SessionProjection`, `ProjectedMessage`, `ProjectedRole` — frontend UI projection types with no Rust serde equivalent
- `TraceEntryData`, `TraceEntryKind`, `TraceEntryStatus` — trace store UI state types
- `SessionMeta`, `ProfileDetail` — already covered by tauri-specta command return types
- ID newtypes (`WorkspaceId`, `SessionId`, `TaskId`, `AgentId`) — serialize as `string`, no benefit from dedicated types

### Consumer refactor

Remove all `const typed = p as { type: "Xxx"; ... }` patterns from:

- `useTauriEvents.ts` — rely on `switch` discriminated union narrowing
- `useTraceStore.ts` — same
- `session.ts` — same
- `taskGraph.ts` — import generated `TaskSnapshot` type

### Helper types (hand-written, import from generated)

- `ExtractPayload<T>` — extract a specific variant by its `type` tag
- `EventPayloadHandlers<R>` — exhaustive handler map
- `PartialEventPayloadHandlers<R>` — partial handler map

## Architecture

### Approach: Extend tauri-specta with event type registration

Use the same specta infrastructure already in the project. Add `specta` as an optional dependency to `agent-core` and `agent-memory`, gated behind a `specta` feature flag. `agent-gui-tauri` enables these features and registers the event types in its specta builder.

### Generation pipeline

1. `just gen-types` runs two binaries:
   - `export-specta` → `generated/commands.ts` (existing, unchanged)
   - `export-events` → `generated/events.ts` (new)
2. `just check-types` becomes: run `gen-types`, then `git diff --exit-code` on `generated/`
3. CI `type-sync` job updated similarly

### File structure

```
crates/agent-core/
  Cargo.toml          ← add specta optional dep + feature
  src/events.rs       ← cfg_attr specta::Type on EventPayload, DomainEvent, PrivacyClassification
  src/task_types.rs   ← cfg_attr specta::Type on AgentRole, TaskState
  src/facade.rs       ← cfg_attr specta::Type on TaskSnapshot, TaskGraphSnapshot
  src/ids.rs          ← cfg_attr specta::Type on ID types + custom Type impl mapping to String

crates/agent-memory/
  Cargo.toml          ← add specta optional dep + feature
  src/memory.rs       ← cfg_attr specta::Type on MemoryScope

apps/agent-gui/src-tauri/
  Cargo.toml          ← enable agent-core/specta + agent-memory/specta features
  src/specta.rs       ← register event types in builder
  src/bin/export_specta.rs  ← unchanged
  src/bin/export_events.rs  ← NEW: export event types binary
  src/lib.rs          ← register export_events binary

apps/agent-gui/src/
  generated/
    commands.ts       ← existing, unchanged
    events.ts         ← NEW: generated by specta
  types/
    index.ts          ← refactored: re-export from generated, keep UI types
    events-helpers.ts ← NEW: ExtractPayload, EventPayloadHandlers
    trace.ts          ← unchanged
  composables/
    useTauriEvents.ts ← refactored: remove as assertions
    useTraceStore.ts  ← refactored: remove as assertions, import from generated
  stores/
    session.ts        ← refactored: remove as assertions
    taskGraph.ts      ← refactored: import TaskSnapshot from generated

scripts/check-types.sh  ← DELETED
justfile                ← updated gen-types and check-types
.github/workflows/ci.yml ← updated type-sync job
```

## Key design decisions

1. **Feature-gated specta in agent-core** — Zero cost when the feature is off (all other crates). Same pattern can be used by TUI if needed later.

2. **Separate export-events binary** — Keeps command type generation and event type generation independent. Each has its own export path and can be run/verified independently.

3. **ID types map to `string`** — specta `Type` impl for `WorkspaceId` etc. maps them to TypeScript `string` rather than opaque branded types. This matches the current hand-written types.

4. **MemoryScope derived via specta** — Even though it's in agent-memory, the consistency of having all event-related types generated from Rust is worth the small optional dep.

5. **Helper types are hand-written** — `ExtractPayload`, `EventPayloadHandlers` are TypeScript utility types that don't correspond to Rust types. They belong in a hand-written file.

## Risk and mitigations

| Risk                                                               | Mitigation                                                                                                                                                                      |
| ------------------------------------------------------------------ | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| specta `tag = "type"` serde output doesn't match current TS format | Verify with PoC before full implementation; EventPayload already uses `#[serde(tag = "type", rename_all = "PascalCase")]` which is exactly specta's discriminated union pattern |
| Optional specta dep causes build issues in other crates            | Feature flag is only enabled in agent-gui-tauri; CI builds all crates without the feature                                                                                       |
| ID newtypes need custom specta::Type impl                          | Specta provides `impl_type!` macro for custom mapping; tested pattern in tauri-specta ecosystem                                                                                 |
