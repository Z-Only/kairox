# TUI/GUI Parity Matrix

This matrix is the working checklist for keeping TUI parity work scoped. It maps GUI generated commands, settings panes, Playwright E2E specs, and Tauri Pilot scenarios to the TUI command, command-palette, and terminal-harness surfaces that should cover the same user capability.

Run `bun run lint:parity-matrix` after adding or removing any generated GUI command, settings pane, GUI E2E spec, or pilot scenario. The check only verifies that the surface is listed here; the row owner still decides whether parity is `covered`, `partial`, `gap`, or `gui-only`.

Status meanings:

- `covered`: TUI has a user-visible surface and a parity smoke, terminal harness, app logic, or keybinding test.
- `partial`: TUI has the core capability, but GUI has richer forms, desktop shell affordances, or extra detail that still needs a narrower follow-up.
- `gap`: no matching TUI surface or harness coverage exists yet.
- `gui-only`: intentionally no terminal equivalent.

- Layered TUI test guide: `crates/agent-tui/tests/README.md`.

When adding a GUI feature:

1. Add the generated command, settings pane, E2E spec, or pilot scenario to this matrix.
2. Decide whether the TUI target is `covered`, `partial`, `gap`, or `gui-only`.
3. If it is not `gui-only`, add or update one of the TUI parity tests above before marking it `covered`.
4. Run `bun run lint:parity-matrix` and the focused TUI parity command listed in the affected row.
